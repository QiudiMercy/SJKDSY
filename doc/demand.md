>

---

## 一、项目概述

《带我玩》是一款 AI 驱动的成都一日游模拟 Web 游戏。用户扮演"导游"，从成都东站接虚拟角色**小爱**，通过自然语言对话决策行程，在真实地理数据上体验一天的游玩。系统通过 LLM Function Calling 裁定行为结果，最终给出评分结算。

**核心玩法循环**：用户发送消息 → 小爱回复（拟人化分段） → 系统裁判裁定行为结果 → 状态更新 → 地图刷新 → 循环，直至触发结束条件。

---

## 二、系统架构

沿用现有 Controller - Service - Agent - Model 四层架构：

```
api/        → 路由控制层，仅做请求分发
services/   → 业务逻辑层，协调 Agent、State、Model
agent/      → 智能体层，LLM 调用与 Function Calling
states/     → 游戏运行时状态封装
models/     → SQLAlchemy ORM
schemas/    → Pydantic 请求/响应模型
tools/      → Function Calling 工具定义 + 裁定器
core/       → 配置 + 统一响应格式
```

**技术栈**：FastAPI + SQLAlchemy + SQLite + OpenAI 兼容 API + 百度地图静态图 API

## 三、后端需求

### 3.1 智能体系统

系统中有**两个智能体**，均抽象为数据库表记录，方便后续扩展更多虚拟角色。

#### 智能体定义表 `agent_definitions`

| 字段 | 类型 | 说明 |
|------|------|------|
| agent_id | String | 角色唯一 ID，如 `"xiaoai"`, `"referee"` |
| role_name | String | 角色名称，如 `"小爱"`, `"裁判"` |
| system_prompt | Text | 角色定义（含语言风格、行为规范） |
| output_spec | Text | 输出规范 |
| tools_list | JSON | 可调用的 Function Calling 工具列表 |

#### 小爱 Agent（虚拟角色）

- **角色定位**：从外地来成都的年轻女性，第一次在成都游玩，对成都充满好奇。性格活泼可爱，会有情绪变化（开心、疲惫、生气）。
- **语言风格**：口语化、亲切、带有适度情绪表达（如"哇这个地方好漂亮！""走得好累哦..."），写入 `system_prompt`。
- **拟人化消息**：回复时，将一大段话拆分为多段短消息（每段 20-50 字），通过多次调用 `send_message` 工具逐段发送，模拟真人打字/思考节奏。
- **Function Calling 工具**：
  - `send_message` — 发送一段短消息到前端（拟人化分段发送的核心工具）
  - `update_status` — 请求修改游戏状态（余额/体力/心情/饱食度/时间/位置），由裁判裁定后生效
  - `search_poi` — 搜索成都 POI 地点
  - `plan_route` — 规划移动路线

#### 裁判 Agent（系统裁定）

- **角色定位**：游戏系统的裁判，负责裁定小爱的状态变更请求是否合法，向用户发送系统通知消息，决定行为结果。
- **职责**：
  - 拦截小爱的 `update_status` 调用，验证数值合法性（如余额不足时阻止扣款）
  - 应用合法的状态变更
  - 必要时向用户发送系统消息（如"余额不足，无法打车"）
  - 检测游戏结束条件
- **实现方式**：基于规则的裁定器（`Referee` 类），非 LLM 驱动。保留现有 `tools/refereetools.py` 的 `validate_and_apply` 模式。

### 3.2 游戏状态管理

#### 状态参数（5 项）

| 参数 | 范围 | 初始值 | 说明 |
|------|------|--------|------|
| money | 0~1000 | 1000 | 余额（元），归零则游戏结束（坏结局） |
| time | 08:00~22:00 | 08:00 | 游戏内时间，到达 22:00 游戏结束 |
| stamina | 0~100 | 100 | 体力值，归零则小爱累倒（坏结局） |
| mood | 0~100 | 100 | 心情值，归零则小爱生气离场（坏结局） |
| fullness | 0~100 | 100 | 饱食度，影响小爱的状态描述，但不直接触发游戏结束 |

#### 饱食度机制

```
初始值 100（早上刚到，吃饱了）
    ↓ 随时间线性衰减
12:00 降至最低点（约 20），11:30-12:30 为午餐适宜窗口
    ↓ 午餐后恢复（若吃饭则快速回满，不吃饭缓慢回至 70）
18:00 降至最低点（约 20），17:30-18:30 为晚餐适宜窗口
    ↓ 晚餐后恢复（若吃饭则快速回满，不吃饭缓慢回至 70）
22:00 游戏结束
```

- **衰减速度**：约 6.7 点/小时（100 → 20 用 12 小时，即上午 8 点到中午 12 点约降 80 点）
- **用餐恢复**（吃饭行为触发）：恢复至 100（快速，模拟吃饭）
- **自动恢复**（不吃饭）：用餐窗口过后自动恢复至 70（缓慢，模拟身体自行调节）
- **影响**：饱食度低时，小爱可能在对话中抱怨饿；但不会因此游戏结束

#### 状态变更流程

```
小爱调用 update_status(money, stamina, mood, fullness, time, location)
    → 裁判 validate_and_apply()
        → 合法：应用变更 + 返回 applied_status
        → 不合法：拒绝 + 返回 system_reply 说明原因
    → 检查游戏结束条件
    → 保存状态到数据库
```

#### 游戏结束条件（不变）

- 时间到达 22:00（Good End）
- 余额归零（Bad End）
- 体力归零（Bad End）
- 心情归零（Bad End）

### 3.3 聊天系统

#### SSE 流式输出

`POST /api/chat/send` 改为 Server-Sent Events 协议：

```
event: message
data: {"segment": "好呀！", "seq": 1}

event: message
data: {"segment": "我们去哪里玩呢？", "seq": 2}

event: status
data: {"updates": {"mood": 102}, "system_reply": "小爱心情不错，心情值+2"}

event: done
data: {"reply": "好呀！我们去哪里玩呢？"}
```

- `message` 事件：小爱的一段短消息
- `status` 事件：状态变更通知（可选，有变更时发送）
- `done` 事件：本轮对话结束，携带完整回复文本

#### 拟人化消息流程

```
1. 用户发送消息
2. 系统组装上下文（游戏状态 + 历史消息 + 小爱 system_prompt）
3. LLM 生成完整回复 + 决定分段方式
4. LLM 通过多次 send_message(function_call) 逐段输出
5. 如有状态变更，LLM 调用 update_status → 裁判裁定
6. 前端逐段展示，营造"打字中"效果
```

### 3.4 地图管理

#### 地图图片生成（新增）

- **路径**：`POST /api/map/image`
- **描述**：根据当前位置和可选的目的地/路线，生成静态地图图片
- **请求参数**：

- **返回**：静态地图图片 URL 或 Base64

### 3.6 评分结算

实现 `/api/game/settle`，评分算法如下：

**评分维度（满分 100 分）**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 余额剩余 | 20% | 剩余金额 / 1000 × 20。省钱有奖励，但不能只省钱不消费 |
| 路线丰富度 | 30% | 访问 POI 数量 / 5 × 30（上限 30）。去了越多地方分越高 |
| 心情均值 | 30% | 整个游戏过程心情平均值 / 100 × 30 |
| 时间利用率 | 20% | (结束时间 - 08:00) / (22:00 - 08:00) × 20。越接近 22:00（时间用满）分越高 |

**评价文案**（根据总分区间）：
- 90~100：「小爱度过了非常充实的一天，你是一个合格的导游！」
- 70~89：「小爱玩得挺开心，但还有更好的安排空间~」
- 50~69：「小爱觉得还行，但有些地方可以改进哦。」
- 30~49：「小爱有点失望，下次要做好攻略呀。」
- 0~29：「小爱非常不开心，这次游玩失败了...」

## 四、前端需求

### 4.1 对话功能

- **消息收发**：对接 `POST /api/chat/send`（SSE 流式），展示小爱分段消息逐条出现的效果
- **历史消息**：页面加载时调用 `GET /api/chat/history` 加载当前游戏的历史对话
- **发送方式**：输入框 + 发送按钮 + Enter 键发送
- **系统消息**：以不同样式（灰色、小字、左侧图标）区分系统裁定消息

### 4.2 状态展示

顶栏实时展示 5 项状态参数，从 `GET /api/game/state` 定时拉取或由 SSE `status` 事件推送更新：

| 展示项 | 图标建议 | 说明 |
|--------|----------|------|
| 金币 | 💰 | 余额（元） |
| 时间 | 🕐 | 当前游戏内时间 |
| 心情值 | 😊 | 0-100 |
| 体力值 | 💪 | 0-100 |
| 饱食度 | 🍔 | 0-100 |

### 4.3 地图

- **展示**：展示后端生成的静态地图图片（调用 `POST /api/map/image`）
- **刷新时机**：位置变更后、导航时、点击 POI 时
- **标注**：当前位置 + 搜索结果的 POI 标注点
- **路线**：导航时在地图上绘制路线

### 4.4 搜索与导航

- **搜索框**：输入关键词搜索 POI，调用 `GET /api/poi/search`
- **默认展示**：无搜索词时展示当前位置附近的推荐 POI
- **POI 详情卡片**：点击 POI 展示详细信息（名称、类型、距离、营业时间）
- **导航操作**：
  - 选择目标 POI → 调用 `POST /api/action/route` 获取三种出行方式预估
  - 展示三种方式（打车/地铁/步行）的费用/耗时/体力消耗
  - 用户选择出行方式 → 发送消息给小爱确认 → 触发移动
  - 移动完成后刷新地图和状态

### 4.5 游戏流程

1. **开始页**：展示游戏简介 + 开始按钮 → 调用 `POST /api/game/start`
2. **游戏主界面**：左侧地图 + 右侧聊天面板 + 顶栏状态 + 搜索功能
3. **游戏进行中**：对话 → 决策 → 状态更新 → 地图刷新 循环
4. **结算页**：游戏结束 → 调用 `POST /api/game/settle` → 展示评分、评价文案、游玩路线回顾
5. **战绩查看**：首页可查看历史游戏记录 → 调用 `GET /api/records/list`

## 六、非功能性需求

- **拟人化**：小爱回复需具备口语化、情绪表达，分段发送模拟真人聊天节奏
- **实时性**：聊天通过 SSE 流式输出，状态更新通过 SSE `status` 事件即时通知前端
- **可扩展**：智能体定义表支持后续新增其他虚拟角色（如"小红"陪玩）
- **数据真实**：使用成都 371K 真实 POI 数据，地图基于真实地理坐标
